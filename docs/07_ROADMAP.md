# Дорожная карта реализации — Радио «Толкай Вода»

10 этапов от пустого репозитория до прод-готовности. Каждый этап заканчивается
**демонстрируемым результатом**.

Ключевая идея: **MVP-first**. Этап 1 = живой радио-плеер на главной. Всё остальное
наслаивается без перепиливания базы.

> **Где смотреть прогресс**: текущий статус всех этапов — в [`STATUS.md`](STATUS.md).
> Внутри каждого этапа ниже — чекбоксы acceptance-пунктов. Закрытие этапа
> сопровождается git-тегом `stage-N-done`.

## Архитектурное решение (важный пункт ТЗ — нагрузка 100 юзеров на 2 ядра / 2 ГБ)

Плеер играет **случайный трек у каждого юзера** (не синхронный эфир), поэтому
HLS/FFmpeg-стрим **не нужен**. mp3 лежат на **S3 Beget** и раздаются через **CDN**
(`ceshocofureled.begetcdn.cloud`). Сайт играет mp3 **напрямую с CDN** — сервер не
трогает аудио-байты вообще. Всю аудио-полосу держит CDN, сервер (2 ядра / 2 ГБ)
обслуживает только лёгкий API блога/прогноза и Telegram-бота.

Стек: **Node.js + Fastify + better-sqlite3**, фронт — HTML/CSS/Vanilla JS,
reverse-proxy **Caddy** (авто-HTTPS), **docker-compose**, деплой **GitHub Actions**.

| # | Этап | Главный артефакт | Зависит от |
|---|---|---|---|
| 0 | Каркас репозитория | `npm run dev` поднимает Fastify, отдаёт пустую страницу | — |
| 1 | Радио-плеер | Кнопка играет случайный трек + анимация дорожки | 0 |
| 2 | S3/CDN треки | Плеер играет реальные 50 mp3 с CDN | 1 |
| 3 | Прогноз дня | Блок на главной с текстом+фото из БД | 0 |
| 4 | Блог | Сетка статей + страница статьи | 0 |
| 5 | Telegram-бот | Сообщение боту → контент на сайте | 3,4 |
| 6 | Контакты/футер | Кастомные иконки соцсетей, email | 1 |
| 7 | Адаптив | Мобильный вид (фон 9:16), оптимизация | 1-6 |
| 8 | Docker/CI/CD | Контейнеры локально + workflow деплоя | 1-7 |
| 9 | Прод-деплой | Сайт открывается на tolkay-voda.ru | 8 |

Проект сдан и принят (этапы 0–9, 2026-06-17). Клиент попросил три доработки.

| 12 | SEO под капотом | robots.txt, sitemap, OG, серверные мета статей, JSON-LD | 4,8 |

> **Изменение архитектурного решения по аудио.** Ранее (раздел выше) было
> зафиксировано «синхронный эфир не нужен, HLS/стрим не нужен — каждый юзер играет
> свой случайный трек с CDN». Клиент изменил требование: нужен **единый эфир как на
> радио** (один поток на всех, вход с текущей точки). Решение по стеку (см. этап 11):
> **Icecast + Liquidsoap**, источник треков — **с CDN** (исходники и их хранение
> остаются на CDN, сервер не держит 50 mp3 на диске), один энкод 128k MP3 → раздача
> через Icecast. Под 2 ядра / 2 ГБ это лёгкая нагрузка (один декод + один энкод +
> fan-out байтов). Раздача слушателям идёт с аплинка сервера; для нишевой аудитории
> (десятки–сотни) это незаметно, апгрейд на HLS+CDN-релей — задел на будущее, если
> аудитория вырастет до тысяч. Старый «джукбокс» остаётся **фолбэком** при недоступном
> потоке.

## Этап 11 — Живой эфир (Icecast + Liquidsoap)

**Цель**: единый синхронный поток на всех слушателей, вход с текущей («live») точки — как на настоящем радио.

**Архитектура**: Liquidsoap тянет треки **с CDN** (плейлист CDN-URL из `tracks.json`),
shuffle + crossfade + метаданные → один **128k MP3** → **Icecast** раздаёт всем.
Caddy проксирует `/stream` и now-playing. Джукбокс (`radio.js`) — фолбэк.

**Что делаем**:
- `docker-compose.yml`: сервисы `icecast` + `liquidsoap` (общий конфиг, source-пароль из `.env`).
- Liquidsoap-скрипт: плейлист CDN-URL, `randomize`, `crossfade`, метаданные now-playing,
  `output.icecast` (mp3 128k); фолбэк-тишина/джингл если плейлист пуст.
- Скрипт генерации плейлиста из `tracks.json` (CDN-URL списком).
- Caddy: `reverse_proxy /stream` → `icecast:8000`, эндпоинт now-playing.
- API now-playing: лёгкий JSON `{title, artist}` (проксирование/парсинг Icecast `status-json.xsl`), с кешем.
- `radio.js`: основной режим — `<audio src="/stream">`, «сейчас играет» из now-playing (поллинг),
  визуализатор на потоке (same-origin → анализатор работает); при недоступном `/stream` —
  автоматический фолбэк на джукбокс.
- Проверка ресурсов (один энкод): CPU/RAM в пределах 2 ядра / 2 ГБ.

**Критерии приёмки** (закрытие → tag `stage-11-done`):
- [x] Два устройства слышат одно и то же в один момент (синхронный эфир)
- [x] Новый слушатель входит в текущую точку трека, а не с начала
- [x] «Сейчас играет» показывает данные потока (у всех треков один бренд-тег; битые ICY-метаданные отфильтрованы → показываем «Радио Толкай Вода»)
- [x] При недоступном `/stream` плеер падает на джукбокс (сайт не «немеет»)
- [x] Источник треков — CDN; 50 mp3 на сервере не хранятся (диск чист)
- [x] `docker compose up` поднимает icecast+liquidsoap, `/stream` отдаёт `audio/mpeg` (проверено на сервере)
- [x] На проде поток стабилен; один 128k-энкод (liquidsoap ~6.7% CPU / 138 МБ, icecast ~2 МБ), запас RAM ~1.2 ГБ

> Стоимость CDN: эфир тянет один источник 24/7 (~80 ГБ/мес ≈ 48 ₽/мес), не зависит
> от числа слушателей (их раздаёт Icecast с сервера, не CDN). Кратно дешевле джукбокса.
